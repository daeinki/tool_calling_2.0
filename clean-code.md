# Clean Code 코딩 가이드

> Robert C. Martin의 *Clean Code: A Handbook of Agile Software Craftsmanship* 의 핵심 원칙을 실무 코딩에 바로 적용할 수 있도록 정리한 가이드 문서입니다.

---

## 0. 왜 깨끗한 코드인가

- 코드를 빨리 짜기 위해 지저분하게 만든다는 것은 착각이다. **빠르게 가는 유일한 방법은 항상 코드를 최대한 깨끗하게 유지하는 것**이다.
- 지저분한 코드는 작성하는 순간부터 생산성을 떨어뜨리고, 결국 마감을 놓치게 만든다.
- 깨끗한 코드는 **한 가지를 잘한다(does one thing well)**. 의도가 명확하고, 의존성이 최소이며, 다른 개발자가 읽고 개선하기 쉽다.
- 깨진 창문 이론: 나쁜 코드 하나가 방치되면 주변 코드도 점점 더 나빠진다. 처음부터 깨끗하게, 그리고 발견하는 즉시 개선한다.

**보이스카우트 규칙**: 코드를 처음 발견했을 때보다 더 깨끗하게 만들고 떠나라.

---

## 1. 의미 있는 이름 (Meaningful Names)

이름은 코드를 읽는 사람에게 의도를 전달하는 가장 강력한 도구다.

### 핵심 규칙

- **의도를 드러내는 이름을 사용한다.** 변수/함수/클래스 이름만으로 존재 이유, 하는 일, 사용법이 드러나야 한다.
  ```cpp
  // Bad
  int d; // elapsed time in days

  // Good
  int elapsedTimeInDays;
  ```
- **그릇된 정보를 피한다.** `accountList`가 실제로 List가 아니라면 `accounts`로 쓴다. 비슷하게 생긴 이름(`XYZControllerForEfficientHandling` vs `XYZControllerForEfficientStorage`)을 피한다.
- **의미 있게 구분한다.** `a1`, `a2`나 `info`, `data`처럼 정보가 없는 불용어(noise word)를 쓰지 않는다.
- **발음하기 쉬운 이름을 쓴다.** `genymdhms` 대신 `generationTimestamp`.
- **검색하기 쉬운 이름을 쓴다.** 매직 넘버 `7` 대신 상수 `MAX_CLASSES_PER_STUDENT`.
- **인코딩을 피한다.** 헝가리안 표기법이나 멤버 접두사(`m_`)는 현대 IDE에서 불필요하다.
- **클래스 이름은 명사**(`Customer`, `Account`), **메서드 이름은 동사**(`postPayment`, `deletePage`)로 짓는다.
- **개념 하나에 단어 하나.** `fetch`, `retrieve`, `get`을 섞어 쓰지 않고 하나로 통일한다.

## 2. 함수 (Functions)

### 작게 만들어라

- 함수는 작아야 한다. **그리고 더 작아야 한다.** 한 화면(약 20줄 이내)을 넘지 않는 것이 이상적이다.
- if/else/while 블록 안의 코드는 한 줄(대개 함수 호출)이어야 한다. 들여쓰기 수준은 1~2단을 넘지 않게 한다.

### 한 가지만 해라

- **함수는 한 가지를 해야 한다. 그 한 가지를 잘해야 한다. 그 한 가지만 해야 한다.**
- 함수 내 모든 문장이 **동일한 추상화 수준**에 있어야 한다. 고수준 개념과 저수준 세부사항을 섞지 않는다.
- 위에서 아래로 이야기처럼 읽히도록 작성한다(**내려가기 규칙, Stepdown Rule**).

### 인수 (Arguments)

- 인수는 적을수록 좋다. **0개(무항)가 가장 좋고, 1개(단항), 2개(이항) 순**이다. 3개(삼항)는 피하고, 4개 이상은 특별한 이유가 없는 한 쓰지 않는다.
- **플래그 인수(boolean)를 피한다.** 함수가 두 가지 일을 한다는 신호다. 두 함수로 분리한다.
  ```cpp
  // Bad
  render(true);

  // Good
  renderForSuite();
  renderForSingleTest();
  ```
- 인수가 많아지면 객체로 묶는다: `makeCircle(x, y, radius)` → `makeCircle(center, radius)`.

### 부수 효과를 없애라

- 함수 이름이 말하지 않는 숨은 동작(부수 효과)을 만들지 않는다. `checkPassword()`가 몰래 세션을 초기화하면 안 된다.
- **명령(상태 변경)과 조회(값 반환)를 분리한다.** 함수는 무언가를 하거나 답하거나 둘 중 하나만 한다.

### 오류 처리

- 오류 코드를 반환하는 대신 **예외를 사용한다.** 정상 경로 로직과 오류 처리 로직이 뒤엉키지 않는다.
- try/catch 블록은 별도 함수로 추출한다. 오류 처리도 "한 가지 일"이다.

### 반복하지 마라 (DRY)

- 중복은 모든 악의 근원이다. 같은 코드가 반복되면 추상화해서 한 곳으로 모은다.

## 3. 주석 (Comments)

> "주석은 나쁜 코드를 보완하지 못한다. 주석을 달기 전에 코드를 고쳐라."

- **최선의 주석은 주석을 달지 않을 방법을 찾는 것**이다. 코드 자체로 의도를 표현한다.
  ```cpp
  // Bad
  // 직원이 복지 혜택을 받을 자격이 있는지 검사
  if ((employee.flags & HOURLY_FLAG) && (employee.age > 65))

  // Good
  if (employee.isEligibleForFullBenefits())
  ```

### 좋은 주석

- 법적 주석, 의도를 설명하는 주석, 결과를 경고하는 주석, TODO, 공개 API의 문서(Javadoc/Doxygen).
- 정규식이나 까다로운 결정의 *이유*를 설명하는 주석.

### 나쁜 주석

- 코드를 그대로 반복하는 주석, 오해의 소지가 있는 주석.
- 의무적으로 다는 주석, 이력을 기록하는 주석(버전 관리 시스템이 할 일).
- **주석 처리된 코드.** 절대 남기지 말고 삭제한다. 버전 관리가 기억해 준다.

## 4. 형식 맞추기 (Formatting)

- 형식은 의사소통이다. 팀이 **하나의 일관된 규칙**에 합의하고 모두가 따른다.
- **세로 형식**: 개념적으로 연관된 코드는 가까이 둔다. 변수는 사용하는 곳 가까이에 선언한다. 한 함수가 다른 함수를 호출하면 세로로 가깝게, 호출하는 함수를 먼저 배치한다.
- **신문 기사처럼**: 위쪽에 고수준 개념과 알고리즘, 아래로 갈수록 세부사항.
- **가로 형식**: 한 줄은 짧게(대략 100~120자 이내). 연산자 주위 공백으로 우선순위를 드러낸다.
- 빈 줄로 개념을 분리하고, 같은 개념은 붙여 둔다.

---

## 5. 객체와 자료구조 (Objects and Data Structures)

- **객체**는 데이터를 숨기고 동작을 노출한다. **자료구조**는 데이터를 노출하고 별다른 동작이 없다. 둘을 명확히 구분한다.
- **디미터 법칙(Law of Demeter)**: 객체는 자신이 다루는 객체의 속사정을 몰라야 한다. 기차 충돌(`a.getB().getC().doSomething()`)을 피한다.
- 잡종 구조(객체도 자료구조도 아닌)를 만들지 않는다. 양쪽의 단점만 갖게 된다.

---

## 6. 오류 처리 (Error Handling)

- **오류 코드보다 예외를 사용한다.**
- **Try-Catch-Finally 문으로 시작한다.** 예외가 발생할 코드를 작성할 때 try 블록부터 작성하면 트랜잭션 범위가 명확해진다.
- 예외에 **의미 있는 정보(맥락)를 담는다.** 오류가 난 위치와 원인을 알 수 있게 한다.
- **null을 반환하지 말고, null을 인수로 넘기지 않는다.** 빈 컬렉션이나 특수 케이스 객체(Special Case Pattern)를 반환한다.

---

## 7. 경계 (Boundaries)

- 외부 라이브러리/API의 경계를 깔끔히 관리한다. 외부 코드는 직접 노출하지 말고 **래퍼(wrapper)로 감싸** 우리 코드에 맞는 인터페이스로 만든다.
- **학습 테스트(learning test)**: 외부 라이브러리 사용법을 테스트로 익히고, 버전 업그레이드 시 호환성을 검증한다.
- 아직 존재하지 않는 코드에 의존할 때는 우리가 원하는 인터페이스를 먼저 정의하고 어댑터로 연결한다.

---

## 8. 단위 테스트 (Unit Tests)

### TDD 세 가지 법칙

1. 실패하는 단위 테스트를 작성하기 전에는 제품 코드를 작성하지 않는다.
2. 컴파일은 되지만 실행이 실패하는 정도로만 단위 테스트를 작성한다.
3. 현재 실패하는 테스트를 통과할 정도로만 제품 코드를 작성한다.

### 깨끗한 테스트

- 테스트 코드도 제품 코드만큼 중요하다. **가독성이 가장 중요하다.**
- **테스트당 개념 하나, assert는 최소로.**
- **F.I.R.S.T 원칙**:
  - **Fast** — 빨라야 자주 돌린다.
  - **Independent** — 테스트끼리 의존하지 않는다.
  - **Repeatable** — 어느 환경에서나 반복 가능해야 한다.
  - **Self-Validating** — 성공/실패가 boolean으로 자동 판별돼야 한다.
  - **Timely** — 제품 코드 직전에 적시에 작성한다.

---

## 9. 클래스 (Classes)

- 클래스는 작아야 한다. 크기는 줄 수가 아니라 **책임의 수**로 잰다.
- **단일 책임 원칙(SRP)**: 클래스나 모듈을 변경할 이유는 단 하나여야 한다.
- **응집도(cohesion)를 높게 유지한다.** 메서드들이 클래스의 인스턴스 변수를 많이 공유할수록 응집도가 높다. 응집도가 떨어지면 클래스를 쪼갠다.
- **변경에 닫혀 있고 확장에 열려 있게(OCP)** 설계한다. 새 기능 추가가 기존 코드 수정이 아닌 확장으로 이뤄지도록 한다.

---

## 10. 시스템 (Systems)

- **관심사를 분리한다.** 객체의 생성(구성)과 사용(실행)을 분리한다. 예: `main`에서 의존성을 조립하고 애플리케이션에 주입한다(DI).
- 시작 단계 로직과 런타임 로직을 섞지 않는다.
- 도시를 설계하듯, 적절한 추상화 수준에서 각 영역을 관리할 수 있도록 모듈화한다.

---

## 11. 창발 (Emergence) — 단순한 설계 4규칙

Kent Beck의 단순 설계 규칙을 **우선순위 순**으로 따른다:

1. **모든 테스트를 통과한다.**
2. **중복이 없다.**
3. **프로그래머의 의도를 표현한다.**
4. **클래스와 메서드 수를 최소화한다.**

---

## 12. 동시성 (Concurrency)

- 동시성은 결합을 분리하지만 복잡성을 높인다. 동시성 관련 코드는 **다른 코드와 분리**한다.
- **공유 데이터를 최대한 줄이고**, 가능하면 공유하지 않는 자료(복사본, 스레드 로컬)를 쓴다.
- 임계 영역(`synchronized`/lock)은 최대한 작게 유지한다.
- 동시성 코드는 별도로 철저히 테스트하고, 실패를 무시하지 않는다.

---

## 13. 냄새와 휴리스틱 (Code Smells) — 빠른 체크리스트

코드 리뷰 시 아래 항목을 점검한다.

**일반**
- [ ] 경직성/중복(DRY 위반)이 있는가
- [ ] 죽은 코드, 사용하지 않는 코드가 있는가
- [ ] 추상화 수준이 섞여 있지 않은가

**함수**
- [ ] 인수가 너무 많지 않은가 (3개 이하)
- [ ] 플래그 인수가 있는가
- [ ] 함수가 한 가지 일만 하는가
- [ ] 사용하지 않는 인수/출력 인수가 있는가

**이름**
- [ ] 서술적이고 의도가 드러나는 이름인가
- [ ] 추상화 수준에 맞는 이름인가 (낮은 수준에는 구체적, 높은 수준에는 추상적)

**주석**
- [ ] 코드로 표현할 수 있는데 주석으로 때우지 않았는가
- [ ] 주석 처리된 코드를 남기지 않았는가

**테스트**
- [ ] 경계 조건과 모든 분기를 테스트하는가
- [ ] 테스트가 빠르고 독립적이며 자동 검증되는가

## 요약: 깨끗한 코드 한 줄 정의

> 깨끗한 코드는 **누군가 정성을 들인 코드(code that has been taken care of)** 이다. 단순하고 직접적이며, 잘 쓴 산문처럼 읽히고, 작성자의 의도를 결코 흐리지 않는다.

**작성 시 항상 자문할 것:**
1. 이 이름이 의도를 드러내는가?
2. 이 함수가 한 가지만 하는가?
3. 중복이 있는가?
4. 테스트가 있는가?
5. 다음 사람이 읽고 고치기 쉬운가?
